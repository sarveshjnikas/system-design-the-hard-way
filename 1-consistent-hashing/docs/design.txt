CONSISTENT HASHING

THE PROBLEM
With naive hashing (key % server_count), adding or removing a server invalidates almost
every key mapping. If you have 1,000,000 keys across 4 servers and add a 5th, nearly all
keys move to a different server.

Consistent hashing fixes this: adding a 5th server moves only ~200,000 keys.
The other 800,000 stay exactly where they are.


BASIC APPROACH
1. Wrap the hash space into a ring (0 to 2^32 - 1)
2. Hash each server → place it on the ring
3. To find a key's server: hash the key → go clockwise → first server encountered owns it

When a server is added: only the keys between it and the previous server on the ring move.
When a server is removed: only its keys move to the next server clockwise.


PROBLEMS WITH BASIC APPROACH
1. Uneven key distribution — servers may cluster on the ring, leaving some with far more keys
2. No guarantee of equal partition sizes across servers


SOLUTION: VIRTUAL NODES
Give each physical server n virtual nodes on the ring.

  server0 → server0_0, server0_1, server0_2, ...
  server1 → server1_0, server1_1, server1_2, ...

Each virtual node is hashed separately and placed on the ring.
Lookup works the same: go clockwise, find the first virtual node, return its parent server.

More virtual nodes → more uniform distribution.
Typical values: 100–200 virtual nodes per server.


IMPLEMENTATION NOTES
- Ring: std::map<uint32_t, string> — sorted by hash position, O(log n) lookup via lower_bound
- Server index: unordered_map<string, vector<uint32_t>> — maps server ID to its virtual node positions
  Used for O(virtual_node_count) removal instead of O(n) full scan
- Hash function: MurmurHash3 (32-bit) — fast, well-distributed, used in production systems
  (Cassandra, Voldemort). Not cryptographic — we don't need that here.
- Collision handling: not implemented. With 32-bit hashes and a small number of virtual nodes,
  collisions are astronomically unlikely. Production systems use 64-bit hashes for extra safety.
